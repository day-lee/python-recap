### 💡 SQL은 결국 "거대한 하나의 가상 테이블을 먼저 만들어두고, 거기서 필터링하고 집계하는 과정"입니다.
- 어떻게 하면 테이블을 여러번 읽지 않고 결과를 가져올 수 있을까?

[1단계] 문제 가독 & 아웃풋 요구사항 노트
   ↓
[1.5단계] ★ 예외/기타 조건 분석 (중복, 정렬, NULL) ★
   ↓
[2단계] 스키마 분석 (PK, FK, 1:N 관계 파악)
   ↓
[3단계] 논리 설계 (Join 유무, 가공 방식 결정)
   ↓
[4단계] ★ 쿼리 작성 (엔진의 순서대로 FROM/JOIN ➔ WHERE ➔ GROUP BY ➔ SELECT) ★



# 문제 풀이형 체크리스트 
======================================================================
[LeetCode SQL Speed-Solving Checklist]
======================================================================
1. OUTPUT & GOAL (출력 및 목표)
   [ ] Granularity: 결과 데이터의 '각 행(Each Row)'이 나타내는 진짜 기준은?
       ※ 헷갈릴 때 보는 3대 패턴 매칭:
       - [ ] 1패턴(단순집계): ~별로 계산하라 ➔ 문장에 있는 기준 그대로 GROUP BY
       - [ ] 2패턴(조건필터): ~한 대상을 찾아라 ➔ 그 대상 자체가 기준 (DISTINCT 주의)
       - [ ] 3패턴(최고/최저): ~별 가장 ~한 사원/상품을 찾아라 ➔ 사원/상품이 기준 (서브쿼리 활용)
   [ ] Columns: 문제에서 요구한 컬럼명(Alias)을 대소문자까지 맞췄는가?
   [ ] Order: 정렬 조건이 있는가? (ASC / DESC 확인)

2. TABLE RELATION (테이블 관계) - 베이스 테이블 만들기 
   [ ] JOIN 필요성 체크: 
       - [ ] "이 두 테이블이 동등한 관계인가, 아니면 한쪽이 다른 쪽을 필터링하기 위한 도구인가?"
       - [ ] YES: 출력할 컬럼이나 필터링 조건이 서로 다른 테이블에 찢어져 있다.
       - [ ] YES: 내 데이터 안에서 이전/이후 행과 비교해야 한다. (Self-JOIN 필요)
       - [ ] NO: 하나의 테이블 안에서 필터링과 집계(GROUP BY)로 해결 가능하다.
   [ ] JOIN Type: INNER JOIN인가, 데이터 유실 없는 LEFT JOIN인가?
   [ ] JOIN Key: 두 테이블을 연결할 매개체 컬럼은 무엇인가?

3. FILTER & FUNCTIONS (필터 및 함수)
   [ ] WHERE vs HAVING: 집계 전 필터인가(WHERE), 집계 후 필터인가(HAVING)?
   [ ] 조건에 맞는 핵심 문법 매칭:
       - [ ] 여러 값 중 하나 매칭 -> IN ('A', 'B') 또는 IN (서브쿼리)
       - [ ] 특정 문자열/패턴 포함 -> LIKE '%text%'
       - [ ] 이상/이하 범위 지정 -> BETWEEN A AND B (또는 >=, <=)
       - [ ] 빈 값(결측치) 제외/수집 -> IS NULL / IS NOT NULL
   [ ] Functions: 윈도우 함수(DENSE_RANK 등)나 날짜/문자열 가공 함수가 필요한가?

4. EDGE CASE (리트코드 단골 함정)
   [ ] Duplication: COUNT인가, COUNT(DISTINCT)인가? (중복 제거 필수 여부)
   [ ] NULL Handling: 데이터가 없을 때 0이나 'None' 처리를 해야 하는가?
   [ ] Ties: 순위 동점자 처리 기준이 있는가? (RANK / DENSE_RANK / ROW_NUMBER)
======================================================================

조인을 해야할지 말아야할지 결정
"내가 지금 다른 테이블에서 새로운 '컬럼 데이터'를 가져와서 화면에 보여줘야 하나?"
YES! 다른 테이블의 필드가 필요하다.
NO! 어차피 같은 테이블에 있는 데이터인데 조건으로 거르기만 하면 됨 ➡️ 이때는 무조건 IN 또는 WHERE 절을 먼저 떠올립니다.
- 조인이 서브쿼리보다는 빠르다. 옵티마이저 최적화가 되어있기 때문. 서브쿼리는 메모리에 임시 테이블을 만들어서 자원소모. 

드라이빙/베이스 테이블 결정
- 누락이 되면 안되는 자료가 어느 테이블에 있는가?

조인 테이블 결정 
- 더 필요한 데이터가 있는가? 
- 로우 개수가 그대로일까, 늘어날까?


explain 실행계획 
type(const, eq_ref, ref, all), key(possible_keys, key), rows(행수)

 *    - [ ] 매칭 실패 시 원본 유실 방지를 위한 [ LEFT JOIN ] 필요 여부 확인

 *    - [ ] 대량 데이터에서 행마다 반복 실행되는 [ 상관 서브쿼리 ] 존재 여부
